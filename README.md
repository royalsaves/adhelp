# Identity Management Portal (Refactoring Record)

해당 콘솔은 재직 중 임직원들의 빈번한 패스워드 변경 요청, 계정 초기화 요청, VPN QR 재생성 요청을 줄이기 위해 만들었던 사내 계정 지원 도구입니다. 자주 들어오는 문의는 셀프서비스 형태로 처리하고, 추가 확인이 필요한 요청은 Slack으로 기록을 남기도록 하여 SRE/DevOps의 반복 작업을 줄이는 데 목적이 있었습니다. 팀의 반복 작업을 줄일 수 있었고, SRE 업무에 더 집중할 수 있었습니다.

## 1. 문제 배경

기존 코드는 Flask 엔드포인트 내부에 아래 요소가 함께 들어가 있었습니다.

- HTTP 요청 처리
- AWS SDK 호출
- LDAP 비밀번호 검증
- 이메일 발송
- 알림 전송
- 보조 자동화 호출

이 구조에서는 아래 문제가 발생합니다.

- 단위 테스트 어려움
- 외부 연동 교체 시 영향 범위 확대
- 공개 저장소 전환 시 내부 도메인 및 운영 맥락 분리 어려움

## 2. 변경 목적

이번 정리는 기능 추가보다 아래 항목에 목적이 있었습니다.

- 결합도 분리
- 실행 환경 단순화
- 공개 저장소 전환 대응
- 실제 인프라 없이도 동작 가능한 데모 환경 구성

## 3. 구조 변경

백엔드는 헥사고날 구조로 다시 나눴습니다.

```text
frontend -> flask api -> application service -> domain port -> adapter
```

각 레이어의 역할은 아래와 같습니다.

- `domain`: 모델과 포트 정의
- `application`: 유스케이스 조합
- `adapters`: 외부 연동 구현
- `api.py`: HTTP 엔트리포인트

이 구조에서는 서비스 레이어가 외부 SDK를 직접 참조하지 않고, 포트 인터페이스를 통해 adapter를 사용합니다.

## 4. 디렉토리 구성

```text
.
├── backend/
│   ├── app.py
│   └── ad_console/
│       ├── api.py
│       ├── app.py
│       ├── config.py
│       ├── domain/
│       ├── application/
│       └── adapters/
├── frontend/
│   └── src/
├── Dockerfile
└── docker-compose.yml
```

## 5. 실행 방식

컨테이너 실행 기준으로 정리했습니다.

```bash
docker compose up --build
```

접속 주소는 아래와 같습니다.

```text
http://127.0.0.1:5001
```

frontend는 Node stage에서 빌드하고, 최종 Python 컨테이너에서 Flask가 정적 파일을 함께 서빙합니다.

## 6. demo / live 모드

실행 모드는 `APP_MODE` 기준으로 분리했습니다.

### demo

- 메모리 저장소 사용
- 외부 연동 대신 로그 출력
- 고정 AI 응답 사용

공개 저장소와 로컬 검증을 위한 모드입니다.

### live

- 실제 외부 연동 adapter 사용
- 같은 서비스 레이어 유지

실제 환경 연동을 위한 모드입니다.

## 7. 운영 고려사항

### 민감정보 처리

- `.env`는 커밋하지 않습니다.
- 공개 저장소에는 `.env.example`만 유지합니다.

### 공개 저장소 전환

아래 항목은 일반화했습니다.

- 회사 식별자
- 내부 도메인
- 내부 운영 절차 문구

### 확장성

다음 변경은 서비스 레이어 수정 없이 adapter 교체 수준에서 대응할 수 있도록 정리했습니다.

- 알림 채널 추가
- 다른 인증/디렉터리 연동 추가
- 저장소를 RDB로 교체
- 관리자 인증 계층 추가
